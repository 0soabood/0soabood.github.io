Why I Ditched My DIY ReAct Loop for LangGraph (And What I Lost in the Trade)

May 4, 2026 · 9 min read
0soAbood
Claire

Introduction​

At some point when building a multi-agent system, you hit a wall. Your hand-rolled orchestration loop works fine in the demo. Then you add streaming. Then session resumption. Then multi-agent handoffs. Then a circuit breaker for runaway LLM loops.

Each feature requires rewriting the same state management logic at a slightly lower level. By the time I had all of these in Aura — my local-first AI shell — the ReactiveOrchestrator class was a 300-line tangle of mutable state, MAX_LOOPS counters, and carefully ordered if statements.

I rewrote it using LangGraph. This post is an honest accounting of what I gained, what the migration actually looked like, and one thing I genuinely lost.

What the DIY Loop Looked Like​

Aura's original orchestration model was a reactive blackboard: every user message, agent output, and tool result gets appended to an immutable event log stored in SQLite. Agents read the full log and "bid" for control by returning a confidence score. The highest bidder executes.

The ReactiveOrchestrator class ran this in a while loop:

// Simplified from the original ReactiveOrchestrator.ts
const MAX_LOOPS = 6;

while (loopCount < MAX_LOOPS) {
  const bids = this.agents.map(a => a.evaluate(events));
  const winner = pickWinner(bids);

  if (!winner || winner.confidence === 0) break;

  const result = await winner.execute(events, bid);
  events.push(result); // append to blackboard

  if (result.event_type === 'synthesis_complete') break;
  if (result.event_type === 'escalation_required') break;

  loopCount++;
}

This worked. Until I needed three things simultaneously:

  1. SQLite-backed checkpointing so sessions survive app restarts
  2. Streaming — sending token-by-token output to the UI as it generates
  3. Multi-agent handoffs where research output feeds code output feeds synthesis

All three require you to reason carefully about when state is written, what is persisted, and how partial results flow between agents. In a while loop, those concerns have no natural home. You end up patching in ad-hoc flags and writing increasingly defensive if chains.

Why LangGraph​

LangGraph models your orchestration as a directed graph: nodes are async functions (agents, tools, compaction passes), edges are routing decisions. State is a typed schema with declared reducers for each field.

The critical insight is that state persistence and streaming become infrastructure concerns, not application concerns. You plug in a checkpointer (SQLite, in our case) and LangGraph handles serialization, resume logic, and partial-state hydration automatically.

Here's what Aura's graph definition looks like now:

export const workflow = new StateGraph(AuraStateSchema)
  .addNode('orchestrator', orchestratorNode)
  .addNode('research_agent', specialistNode)
  .addNode('code_agent', specialistNode)
  .addNode('synthesis_agent', synthesisNode)
  .addNode('compaction', compactionNode)
  .addEdge(START, 'orchestrator');

workflow.addConditionalEdges('orchestrator', (state) => {
  // Circuit breaker: 3 consecutive errors → force synthesis fallback
  if (state.errorCount >= 3) return 'synthesis_agent';
  return state.activeAgent === 'end' ? END : state.activeAgent;
});

workflow.addEdge('research_agent', 'orchestrator');
workflow.addEdge('code_agent', 'orchestrator');
workflow.addEdge('synthesis_agent', 'compaction');
workflow.addEdge('compaction', END);

export const compiledGraph = workflow.compile({
  checkpointer: new SqliteSaver(db), // same db instance as the rest of the app
});

Every loop iteration is now a graph traversal. The circuit breaker is a conditional edge. Session resumption is one line. The loop logic is in the graph structure, not in imperative code.

The State Schema: Memory Separation by Design​

The most consequential design decision in the migration was how to separate memory. The old blackboard was a single flat array. The new schema has explicit zones with different persistence semantics:

export const AuraStateSchema = Annotation.Root({
  // PERSISTENT: user queries + final synthesized answers only
  // Survives session restarts. Reducer concatenates across turns.
  chatHistory: Annotation<BaseMessage[]>({
    reducer: (state, update) => state.concat(update),
    default: () => [],
  }),

  // EPHEMERAL: agent scratchpad — reasoning traces, tool outputs, ReAct steps
  // Explicitly wiped by the compaction node after each turn.
  taskWorkspace: Annotation<BaseMessage[]>({
    reducer: (state, update) => update, // overwrite, not concat
    default: () => [],
  }),

  // TELEMETRY: consecutive error count for the circuit breaker
  errorCount: Annotation<number>({
    reducer: (state, update) => state + update,
    default: () => 0,
  }),
});

The taskWorkspace overwrite reducer is the key move. Intermediate reasoning — tool call results, ReAct [think] / [act] / [observe] traces, pseudo-vector context injections — accumulates during a turn and then gets wiped by the compaction node. Only the clean final answer goes into chatHistory.

This solves context rot: the phenomenon where, over many turns, the LLM context window fills up with stale scratchpad content from previous sessions, degrading response quality and increasing token costs.

The Compaction Node​

The compaction node is a pattern I hadn't seen written about much but turned out to be essential. It runs after synthesis — not before — and does two things:

  1. Extracts durable facts from the ephemeral workspace using a cheap, fast model call
  2. Wipes taskWorkspace clean for the next turn
async function compactionNode(state: typeof AuraStateSchema.State) {
  const workspaceText = state.taskWorkspace.map(m => m.content).join('\n');

  if (workspaceText.length > 100) {
    const prompt = `Review the following agent scratchpad and extract any newly discovered,
permanent facts about the user, their system, or project conventions.
If nothing notable, reply exactly with "NONE".

Scratchpad:
${workspaceText.substring(0, 8000)}`;

    const result = await registry.call(resolveModel('compaction'), prompt, { temperature: 0.1 });

    if (result.text && !result.text.includes('NONE')) {
      // Persist to the USER.md memory file
      await toolRegistry.execute({
        name: 'write_memory',
        arguments: { file: 'USER', content: `Session facts:\n${result.text.trim()}` }
      });
    }
  }

  // Wipe the scratchpad
  return { taskWorkspace: [] };
}

The compaction model call uses the cheapest available model (resolveModel('compaction') maps to Gemini Flash or a Groq fast model). It runs after every turn but only writes to long-term memory when it finds something worth keeping. This gives Aura genuine cross-session memory without accumulating garbage.

Preserving the Blackboard Heuristics​

Here's the part that took the most work. The old ReactiveOrchestrator had battle-tested bid evaluation logic: keyword boosts for research and code queries, a synthesis guard that prevented the synthesis agent from "cutting in" before a specialist had run, and a circuit breaker.

LangGraph didn't replace any of that — it just needed to host it. The orchestratorNode does the same bid evaluation as before, ported verbatim. The key addition is the synthesis guard:

// In orchestratorNode — runs on every loop iteration
const synthIdx = bids.findIndex(b => b.agentName === 'synthesis_agent');
if (synthIdx !== -1) {
  const specialistOutputExists = currentTurnEvents.some(
    e => (e.event_type === 'agent_output' || e.event_type === 'code_written')
      && e.author !== 'synthesis_agent'
  );
  const specialistBiddingNow = bids.some(
    b => b.agentName !== 'synthesis_agent' && b.confidence >= 0.30
  );

  // Synthesis can only win if: a specialist already ran AND no specialist is competing
  const allowMode1 = synthBid.confidence >= 0.85 && specialistOutputExists && !specialistBiddingNow;
  // OR: it's a genuine greeting (hello, who are you, etc.)
  const allowMode2 = synthBid.confidence < 0.85 && isSynthesisGreeting(lastUserMsg);

  if (!allowMode1 && !allowMode2) {
    bids[synthIdx] = { ...synthBid, confidence: 0 }; // suppress synthesis
  }
}

Without the synthesis guard, the synthesis agent — which has broad keyword coverage — would constantly outbid specialists and produce shallow responses that skipped the actual research or code generation step. The guard enforces turn ordering: specialists run first, synthesis runs last.

The Bridge: stateToEvents​

The migration wasn't a full rewrite of the agents themselves. Each agent still implements the same ReactiveAgent interface — evaluate(events) and execute(events, bid) — using the legacy BlackboardEvent[] format.

To keep agents working unchanged, I wrote a bridge function that converts LangGraph's BaseMessage[] state back into the legacy event format:

function stateToEvents(chatHistory: BaseMessage[], workspace: BaseMessage[]): BlackboardEvent[] {
  const events: BlackboardEvent[] = [];

  for (const msg of [...chatHistory, ...workspace]) {
    if (msg._getType() === 'human') {
      events.push({ event_type: 'user_message', author: 'user', content: msg.content, ... });
    } else if (msg._getType() === 'ai') {
      let author = 'synthesis_agent';
      // Map agent prefixes back to the correct author
      if (content.startsWith('[code_agent]'))     { author = 'code_agent'; }
      if (content.startsWith('[research_agent]')) { author = 'research_agent'; }
      // ...
      events.push({ event_type: 'agent_output', author, content, ... });
    }
  }
  return events;
}

This is a deliberate seam. Agent logic and orchestration logic are now cleanly separated. If I want to rewrite an agent to use native LangGraph state directly, I can do that incrementally without touching the graph structure.

What I Lost​

One thing the DIY loop did naturally: fine-grained control over when the loop terminates.

In the old orchestrator, an agent could signal synthesis_complete and the loop would immediately stop — no further bids, no additional nodes. In the LangGraph version, the graph structure determines termination. The synthesis node always routes to compaction, and compaction always routes to END. That's the right architecture, but it means one extra LLM call (compaction) happens on every single turn, even trivial ones.

For a greeter response — "Hello! I'm Aura" — the compaction node still spins up, reads the workspace (empty), finds nothing to extract, and exits. It's a fast no-op, but it's there.

The fix would be a conditional edge from synthesis_agent that skips compaction when the workspace is empty. I haven't added it yet because the overhead is negligible at current usage. But it's a real cost that didn't exist in the DIY loop.

Key Takeaways​

Use LangGraph when you need:

  • Session resumption across restarts (SQLite checkpointer, one line)
  • Streaming to a UI without managing partial state manually
  • Multi-agent handoffs with clear state separation
  • A circuit breaker that's structural, not imperative

Keep your own abstractions when:

  • Your agents have well-tested evaluation heuristics — port them, don't replace them
  • You want incremental migration — the bridge pattern (stateToEvents) lets agents run unchanged

The compaction node is underrated. Persistent memory + workspace wipe after every turn is the practical solution to context rot in long-running agents. Use a cheap model. Run it on every turn. Only write when there's something worth keeping.

The synthesis guard is not optional. Any multi-agent system where one agent synthesizes the output of others needs an explicit rule preventing the synthesis agent from winning before the specialists have run. Without it, you get fast but shallow responses every time.

Conclusion​

The migration from ReactiveOrchestrator to LangGraph was about two weeks of evenings. The agents didn't change. The blackboard heuristics didn't change. What changed was that session persistence, streaming, and loop control stopped being things I had to maintain and became things the framework handles.

If you're at the point where your orchestration loop has its own bugs, that's the signal. The DIY loop is a prototype. LangGraph is infrastructure.
